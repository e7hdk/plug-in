## What this capability adds
- **No hardcoded user-facing strings** — everything is externalized into message catalogs with keys + translator context.
- **Compose, don't concatenate** — ICU MessageFormat for plurals/gender/interpolation.
- **Locale-aware formatting** — dates, numbers, currency, lists, and collation via `Intl`/CLDR; no baked-in 12h/`.`/MM-DD assumptions.
- **Layout tolerance** — ~30–40% text expansion and **RTL** (bidi, mirrored layout/icons).
- **Translation workflow** — extract → translate → merge, with catalogs kept in sync; pseudolocalization in QA.

## Agents it activates
`i18n-engineer`. (Also sharpens `frontend-builder` / `mobile-builder` to externalize strings,
`design` to accommodate expansion + RTL, `growth` to write translation-ready source copy, and `qa`
to run the pseudolocale.)

## Gates / concerns
- No hardcoded user-facing text; no fragment concatenation; locale-aware formatting everywhere.
- Layout survives +30–40% text and RTL; a sensible fallback chain exists.
- New strings ship with a key + translator comment; no orphan/missing catalog keys.

## Role notes
- builders: externalize strings as you add them; use the catalog, never a literal.
- design: leave room for expansion; mirror for RTL.
- growth/content: write source copy that translates cleanly (avoid idioms, embedded variables in awkward spots).
- qa: run pseudolocalization + at least one RTL locale before sign-off.
